Purpose

To provide an overview of the organization of animations in UO, how they are stored, and how they are mapped.

Animations

Ultima Online use a sequence of still images (frames) to represent an animation (much like movie film). The data that represents these images are stored in a data file (anim.mul, anim2.mul,anim3.mul,anim4.mul,anim5.mul). This data is indexed by the corresponding index file (anim.idx,anim2.idx,anim3.idx,anim4.idx,anim5.idx). Each entry in the index file (3, 4 byte words, that represent the offset into the mul file, length of the data, and extra word that is not currently used) points to a data blob in the mul file that represents the frames (usually between 0-10) for a given model, activity, and direction. The data consists of a 256 color palette, number of frames (frames can be empty, contain no data), and the data for each frame. The frames all share the common palette for that entry.

Organization

At the lowest level, one can access a sequence of images that represents a model performing some action in some direction by directly traversing any of the anim idx files. However, one normally wants to logically group a collection of these entries to represet a complete model, and actions. UO uses the concept of "body types" to group the entries, to represent a model (model id), collection of actions (action id), and directions (direction id) for each action. For UO there are: 5 body types, and 5 directions.

There are 5 directions any action for any model can have. The directions are: Down,Down/Left,Left,Up/Left/Up. Each direction corresponds to an entry in the index file, that provides the data for the frames for that action/direction combination.

Structurally, a category can have N actions, as long as one understands that information for a given category. For the current body types, the following table defines the number of allocated actions.

Body Type Actions used Number of IDX entries
Monster 22 110
Sea Monster 9 45
Animal 13 65
Equipment 35 175
Human 35 175

Body type mappings

There are 2048 possible body ids unique (models). Each body id is defined a body type, via the "mobtypes.txt" file. This file defines a type to an id, along with a flag that indicates some information about that model. For model ids that are not defined in this file (or clients before 5.0), body types are defined by model id via the following:
Model Id Range Body Type
Monster 0-199
Animal 200-399
Human/Equipment 400-2408

Mobtypes.txt

This file takes the format:

ID Category Flag

example:
1 MONSTER 0
defines the model id 1, as a body type of "Monster", with no additional flags.
Flags
The flag value in "mobtypes.txt" has the following meaning (for known values):
Body Type Flag value Meaning
Monster A00 Attack actions 4,5,6 replace actions 12,13, and 14
Animal 20 Action 6 is transformed on water to allow animals such as snakes to float on water
Monster 1000 Actions 12, 13 and 14 cast spells
Monster 1000 Actions 7 and 8 are added, and action 12 is used as the summoning animation for the monster
Monster 1000 Action "fly" 19, 20, 21 which are supported, but also 12 and 13 with spell caster
Monster 1000 Action "fly" 19, 20, 21 which are supported, but also 12 with caster
All 10000 Animation is stored in the UOP files
Monster 10008 Animation is stored in the UOP files,unknown what the other bit indicates
Animal 12000 Animation is stored in the UOP files,unknown what the other bit indicates
Human 12000 For "bodysuits", e.g. Lord British. Shows bodysuit and blocks other worn gump art

Action swapping

The client has the ability to swap actions for different different body types. "Anim1.def" defines the actions that are to be substituted when an action is received for Animal body types. "Anim2.def" defines the actions that are to be substituted when an action is received for Character and Equipment body types. This file takes the format:

action id {new action id} unused

example:
13 {5} 0
defines the acton id 13, to be replaced with action id 5.

Swapping Equipment types

For model ids that have the human type, the equipment ids can be swapped to another equipment id (with a hue). In addition, a gump can be specified to be used. The file "Equipconv.def" specifies a given model id, and a given equipment model id, what to substitute for it. This file takes the format:

modelid equipmentid newequipementid gumpid hue

Note: gumpid (If 0, then use equipmentid + 50000, if -1 then use newequipementid + 50000, other numbers are the actual gumpID )e

example:
249 1250 61250 0
replaces for model id 249, the equipmentid 1250 to equipmentid 61250, with no gump id being also 61250 (50000+1250)

Creating/swapping Body ids

The client has the ability to map a model id to an existing model id with a hue. This is performed via the mapping table defined in the "Body.def" file. This file takes the format:

modelid {new model id} hue

example:
11 {28} 1401
defines the model id 11, to be replaced with model 28, hued with 1401.

Overriding the Corpse

The file "Corpse.def" allows one to override the corpse id to be used, with none, or another one with a given hue. This file takes the format:

modelid {new model id} hue

example:
11 {28} 1401
defines the model id 11, to be replaced with model 28, hued with 1401.

Defining what model id comes from what animation file

The file "Bodyconf.def" allows one to define for any model id, what anim # file to use, and what id out of that file to use. This file takes the format:

modelid anim2 anim3 anim4 anim5 anim6(future?)

example:
157 1 -1 -1 -1 -1
defines the model id 15, use model id 1 in anim2.mul.
